# CampusVoice v0.3 self-review and repair prompt

- Review date: 2026-07-13
- Scope: notice versions, evidence claims, impact propagation, atomic migration, verification, group undo, privacy, UI, and final gates
- Status: repair complete; all runnable gates pass, real Compose smoke is environment-blocked by the unavailable Docker Engine

## Evidence-backed findings

The first implementation passed its initial backend and frontend gates, but a second requirement-by-requirement review found safety gaps that tests did not yet prove:

1. v0.3 notice, impact, and migration rows were missing from privacy export, deletion counts, explicit deletion, and post-delete verification.
2. migration execution trusted the preview-time calendar conflict result and did not re-check for conflicts created after preview.
3. impact detection selected every task/event linked to the old document instead of requiring the exact old claim lineage.
4. a reviewed change could be rejected after preview without invalidating its impacts or ready plan.
5. the commit-to-verification window left `applied` and `undo_applied` plans without a retry/recovery path after interruption.
6. execute and undo shared one mutable verification payload, so undo could overwrite the earlier execution receipt.
7. sequential idempotency tests did not prove that two independent concurrent requests cannot both execute or undo a plan.
8. grade normalization did not treat the settings UI value `2024 级` as equivalent to the extracted claim value `2024`.
9. cosmetic Unicode, full-width punctuation, and spacing changes were insufficiently normalized, and astral Unicode evidence offsets needed a direct round-trip test.
10. removed claims and audience de-scoping had no explicit keep/cancel/manual-review recommendation.
11. the radar and detail UI lacked complete applicability, failure, retry, source, database-detail, real timeline, focus, mobile, and verification-failure evidence.
12. users could inspect a seeded/API-created version chain but had no first-class UI for explicit series and successor import.
13. package and API metadata still identified the completed v0.3 feature set as version 0.2.0.
14. the synthetic v1 story mentioned a campus-card requirement but did not create the separate, traceable reminder required by the acceptance story.
15. the first radar UI implementation changed only local button text on the first high-risk click, then auto-issued, advanced, and consumed both server challenge stages on the second click.
16. when browser `sessionStorage` access failed, begin/finish interactions could generate different migration idempotency keys and invalidate the body-bound challenge.
17. the multi-worker Compose path did not inject a shared confirmation secret, so separate Uvicorn processes could generate different HMAC keys and randomly reject challenge issue/advance/consume requests.

## Generated repair prompt

> Starting from the current dirty CampusVoice worktree, preserve all existing security and test behavior. Repair every finding above without resetting, cleaning, committing, pushing, or opening a PR. Make privacy lifecycle handling explicit and user-scoped. Re-check mutable safety conditions inside the execution transaction. Propagate only exact evidence lineage. Invalidate stale plans when review or applicability changes. Make post-commit verification resumable and keep execute/undo receipts separately. Prove execute and group undo concurrency with independent requests or sessions. Normalize real settings values and cosmetic Unicode deterministically while preserving evidence offsets. Represent removed/de-scoped impacts as safe recommendations rather than silent deletion. Complete the responsive UI with source traceability, calendar preview, failure recovery, accessible focus, and explicit version import. Add regression tests for each bug, then rerun Ruff, formatting, mypy, pytest and coverage gates, both pip audits, frontend tests/coverage/typecheck/lint/format/build/audit, Playwright mock E2E, Compose config parsing, the Alembic round trip, and real Compose smoke when a Docker engine is available.
